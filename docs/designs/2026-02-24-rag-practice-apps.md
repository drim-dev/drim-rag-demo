## Обзор

Три самостоятельных RAG-приложения с возрастающей сложностью, каждое из которых демонстрирует концепции из урока по RAG. Студенты клонируют и запускают каждое приложение независимо. Все три написаны на Python с простым веб-интерфейсом.

**Прогрессия данных:** простой текст → структурированная документация → многоисточниковые разнородные данные
**Прогрессия фреймворков:** LlamaIndex (простой) → LangChain (больше контроля) → LlamaIndex (продвинутые функции)
**Прогрессия провайдеров:** OpenAI → Cohere → локальные эмбеддинги (Ollama) + Claude

**Расположение:** поддиректории `practice/rag/` (позже будут вынесены в отдельный репозиторий).

## Приложение 1: «Ask Paul Graham»

### Цель

Студенты получают момент «RAG работает!» примерно в ~50 строках кода.

### Покрывает разделы урока

проблема, архитектура, реализация

### Интерфейс: одностраничное Streamlit-приложение

**Боковая панель — Управление индексацией:**
- Кнопка «Скачать эссе» — запускает скрипт загрузки, показывает прогресс-бар
- Отображение количества документов (например, «Загружено 213 эссе»)
- Кнопка «Построить индекс» — чанкинг, эмбеддинги, сохранение в Chroma
- Статистика индекса после построения: количество чанков, размерность эмбеддингов, затраченное время
- Слайдер размера чанка (256 / 512 / 1024 токенов) — пересоберите индекс, чтобы увидеть эффект
- Слайдер перекрытия чанков (0% / 10% / 20%)

**Основная область — Интерфейс запросов:**
- Поле ввода текста: «Задайте вопрос Полу Грэму»
- Кнопка «Спросить»
- Область ответа с потоковым отображением
- Раскрывающийся раздел «Извлечённые чанки» под ответом:
  - Каждый чанк показывает: название исходного эссе, оценка релевантности (косинусное сходство), предпросмотр текста (первые 200 символов с возможностью развернуть)
  - Чанки отсортированы по оценке релевантности
- Слайдер Top-K (1–10, по умолчанию 5)

### Что изучат студенты

- Загрузка документов и базовый чанкинг (фиксированный размер с перекрытием)
- Генерация эмбеддингов и векторное хранилище
- Поиск по сходству и извлечение top-K
- Дополнение промпта извлечённым контекстом
- Полный цикл базового RAG-конвейера

### Ключевые особенности

- Один индекс, один тип документов
- Стратегия чанкинга по умолчанию (рекурсивный посимвольный)
- Поиск по косинусному сходству
- Простой шаблон промпта («отвечай только на основе контекста»)
- Интерфейс показывает: поле запроса, ответ, извлечённые чанки с источниками

### Предложения по упражнениям

1. Измените размер чанка с 512 на 128 — наблюдайте, как ответы ухудшаются при слишком маленьких чанках
2. Установите top-K в 1 против 10 — увидите компромисс между точностью и полнотой
3. Задайте вопрос о событиях после 2023 года — наблюдайте поведение галлюцинаций vs «Я не знаю» при хорошем шаблоне промпта

## Приложение 2: «Python Docs Assistant»

### Цель

Студенты видят, почему наивный RAG не справляется со структурированными документами, и изучают техники для решения этой проблемы.

### Покрывает разделы урока

варианты использования, продвинутые техники

### Интерфейс: Gradio-приложение с вкладками

**Вкладка 1 — «Индекс»:**
- Кнопка «Скачать документацию Python» с прогрессом
- Статистика источника: количество документов, общий размер
- Выбор стратегии чанкинга: фиксированный размер / по заголовкам (RST) / семантический
- Кнопка «Построить индекс» — показывает прогресс по этапам (парсинг → чанкинг → эмбеддинги → сохранение)
- Статистика индекса: количество чанков по типу документа, распределение полей метаданных (круговая диаграмма: модули stdlib по количеству чанков)

**Вкладка 2 — «Поиск» (основная вкладка):**
- Поле ввода запроса
- **Переключатели метода извлечения (чекбоксы):**
  - Векторный поиск (включён по умолчанию)
  - BM25 поиск по ключевым словам (выключен по умолчанию)
  - Гибридный с RRF (выключен по умолчанию — при включении активируются оба: векторный + BM25)
  - HyDE трансформация запроса (выключена по умолчанию)
  - Реранжирование (выключено по умолчанию)
  - Parent-child извлечение (выключено по умолчанию)
- **Фильтры по метаданным:**
  - Выпадающий список модулей (например, «все», «os», «pathlib», «asyncio», «collections»)
  - Выпадающий список типа документа («все», «учебник», «справочник API», «руководство»)
- Слайдер Top-K (1–20, по умолчанию 5)
- Кнопка «Спросить»

**Отображение ответа:**
- Сгенерированный ответ с потоковым выводом
- Раздел «Источники»: название модуля, путь раздела, ссылка на исходную страницу документации Python
- Аккордеон «Извлечённые чанки»:
  - Каждый чанк: оценка релевантности, метод извлечения (векторный / BM25 / оба), теги метаданных, текст
  - Подсветка пересечения, когда гибридный поиск возвращает один и тот же чанк обоими методами

**Вкладка 3 — «Сравнение»:**
- Один и тот же запрос обрабатывается двумя конвейерами параллельно
- Слева: наивный (только векторный поиск, чанкинг по умолчанию)
- Справа: продвинутый (гибридный + реранжирование + чанкинг по заголовкам)
- Показаны оба ответа и оба набора чанков
- Внизу: сводная таблица — количество пересечений чанков, средняя оценка релевантности, длина ответа

### Что изучат студенты

- Чанкинг с учётом структуры документа (разбиение по RST-заголовкам с сохранением иерархии разделов)
- Гибридный поиск: векторный (семантический) + BM25 (по ключевым словам) с Reciprocal Rank Fusion
- Фильтрация по метаданным (фильтр по модулю, типу раздела: учебник vs справочник API)
- Реранжирование с помощью кросс-энкодера (Cohere rerank или open-source)
- HyDE трансформация запроса (генерация гипотетического ответа, поиск по его эмбеддингу)
- Parent-child извлечение (находим маленький чанк, возвращаем родительский раздел для контекста)

### Ключевые особенности

- Чанки, обогащённые метаданными (название модуля, путь раздела, тип документа)
- Включение/выключение гибридного поиска для наблюдения разницы
- Включение/выключение реранжирования для сравнения качества результатов
- Сравнение бок о бок: наивное vs продвинутое извлечение
- Интерфейс показывает: поле запроса, ответ, извлечённые чанки с оценками релевантности, использованный метод извлечения

### Сравнение с Приложением 1

Наивный подход Приложения 1 не работает с документацией Python — запрос вроде «как открыть файл» возвращает разрозненные результаты из `os`, `io`, `pathlib`, `builtins`. Приложение 2 показывает, как гибридный поиск + фильтрация по метаданным + реранжирование решают эту проблему.

### Примеры запросов (предзаполненный выпадающий список)

- «Как прочитать файл построчно?» — тестирует базовое извлечение
- «В чём разница между os.path и pathlib?» — тестирует кросс-модульное извлечение
- «asyncio.gather timeout» — тестирует поиск по ключевым словам (точное имя API)
- «GIL» — тестирует обработку аббревиатур

### Предложения по упражнениям

1. Сравните чанкинг по заголовкам vs фиксированный размер на запросе «Как работает система импорта Python?» — чанкинг по заголовкам должен вернуть связные разделы
2. Попробуйте «asyncio.Lock» с только векторным поиском vs гибридным — BM25 отлично справляется с точными именами API
3. Включите реранжирование и наблюдайте, как меняется порядок результатов для неоднозначных запросов

## Приложение 3: «K8s Knowledge Base»

### Цель

Студенты создают продакшен-уровневую мульти-индексную RAG-систему с оценкой, мониторингом и несколькими типами данных.

### Покрывает разделы урока

оценка, продакшен, варианты использования (мульти-индекс, поиск по коду, API-спецификации)

### Интерфейс: Streamlit-приложение с 3 страницами

**Страница 1 — «Индекс» (`/index`):**
- Три карточки источников данных:
  - **Документация:** кнопка «Скачать документацию K8s», количество файлов, количество чанков, время последней индексации
  - **Код:** кнопка «Скачать исходный код K8s», количество файлов (Go-файлы), количество извлечённых функций/типов, количество чанков
  - **API-спецификации:** кнопка «Скачать OpenAPI-спецификации», количество эндпоинтов, количество чанков
- Каждая карточка показывает статус индексации независимо
- Кнопка «Переиндексировать изменённые» для каждого источника — инкрементальная индексация по хешам, показывает «3 из 847 файлов изменены, переэмбеддинг 3»
- Кнопка «Переиндексировать всё» с диалогом подтверждения
- Глобальная статистика: общее количество векторов по всем индексам, размер БД, информация о модели эмбеддингов

**Страница 2 — «Чат» (`/`) — главная страница:**
- Чат-интерфейс с потоковыми ответами
- Каждый ответ показывает:
  - Потоковый текстовый ответ от Claude
  - Теги «Источники» под ответом: цветовая кодировка по типу источника (синий = документация, зелёный = код, оранжевый = API-спецификация)
  - Клик по тегу источника разворачивает текст чанка и метаданные
  - Индикатор «Маршрут» — какой индекс(ы) выбрал маршрутизатор (например, «документация + код» или «только API-спецификации»)
- Визуализация маршрутизации запросов:
  - Маленький бейдж с классификацией маршрутизатора (например, «концептуальный → документация», «реализация → код + документация», «API-контракт → спецификации»)
  - Оценка уверенности от LLM-вызова маршрутизатора
- Индикатор семантического кеша: бейдж «Из кеша», когда ответ берётся из кеша, со ссылкой на исходный запрос, заполнивший кеш
- Метрики по каждому сообщению (раскрывающиеся):
  - Разбивка задержки: эмбеддинг (мс) → маршрутизация (мс) → поиск по индексу (мс) → генерация (мс) → итого
  - Использование токенов: токены эмбеддингов, входные/выходные токены генерации
  - Ориентировочная стоимость этого запроса
- Кнопки «палец вверх» / «палец вниз» на каждом ответе
- Поле ввода обратной связи (появляется после «палец вниз»)

**Страница 3 — «Оценка» (`/evaluate`):**
- **Раздел тестового набора данных:**
  - Готовый тестовый набор: 50 троек вопрос-ответ-контекст
  - Табличное представление: вопрос, краткое содержание ожидаемого ответа, тип источника
  - Кнопка «Запустить оценку» — запускает конвейер RAGAS по всем вопросам
  - Прогресс-бар во время оценки
- **Панель результатов:**
  - Четыре карточки метрик: Faithfulness, Answer Relevancy, Context Relevancy, Context Precision — каждая показывает оценку 0–1 с цветовой индикацией (красный < 0.5, жёлтый 0.5–0.8, зелёный > 0.8)
  - Таблица разбивки по вопросам: вопрос, оценка каждой метрики, пройдено/не пройдено
  - Раскрывающиеся строки: показывают извлечённый контекст, сгенерированный ответ, пояснения метрик
  - Анализ ошибок: «5 вопросов не прошли по Faithfulness — основная проблема: галлюцинированные имена namespace в примерах кода»
- **Метрики во времени:**
  - График оценок по запускам оценки (хранятся в БД)
  - Полезно для наблюдения улучшений по мере настройки конвейера студентами
- **Панель стоимости и производительности:**
  - Средняя задержка по этапам (столбчатая диаграмма)
  - Тренд стоимости за запрос (линейный график за последние 100 запросов)
  - Процент попаданий в кеш
  - Сводка обратной связи: соотношение «палец вверх»/«палец вниз»

### Что изучат студенты

- Мульти-индексная архитектура: три отдельных индекса (документация, код, API-спецификации)
- Маршрутизация запросов: LLM-классификатор решает, в каких индексах искать
- Чанкинг с учётом кода (tree-sitter для Go — функции, типы, интерфейсы)
- Чанкинг API-спецификаций (каждый эндпоинт = чанк с методом/путём/запросом/ответом)
- Конвейер оценки RAGAS (faithfulness, answer relevancy, context relevancy)
- Создание набора данных для оценки и измерение базовых показателей
- Семантическое кеширование (косинусное сходство > 0.95 для пропуска LLM-вызова)
- Учёт стоимости за запрос (токены эмбеддингов + LLM)
- Инкрементальная индексация с обнаружением изменений по хешам

### Ключевые особенности

- Три индекса с автоматической маршрутизацией запросов
- Вкладка оценки: запуск метрик RAGAS на тестовом наборе, оценки по каждой метрике
- Панель метрик: разбивка задержки, стоимость за запрос, процент попаданий в кеш
- Инкрементальная индексация: обнаружение изменений по хешам, переэмбеддинг только изменённых файлов
- Обратная связь от пользователя: «палец вверх»/«палец вниз» на ответах, хранится в БД
- Потоковые ответы

### Сравнение с Приложениями 1 и 2

Запрос вроде «как работает kubectl apply?» попадает во все три индекса — документация объясняет концепцию, исходный код показывает реализацию, API-спецификация показывает контракт эндпоинта. Маршрутизатор решает запросить все три, и ответ синтезируется из разных источников. Конвейер оценки доказывает, что мульти-индексный подход набирает более высокие оценки, чем одноиндексный.

### Примеры запросов (предзаполненные)

- «Как работает kubectl apply?» — попадает во все три индекса (документация объясняет, код показывает реализацию, API-спецификация показывает эндпоинты)
- «В чём разница между Deployment и StatefulSet?» — концептуальный, направляется только в документацию
- «Покажи поля спецификации Pod для ограничений ресурсов» — направляется в API-спецификации
- «Как реализован планировщик?» — направляется в код + документацию

### Предложения по упражнениям

## Проектные решения

### Почему всё на Python?

Экосистема инструментов RAG (LlamaIndex, LangChain, RAGAS, привязки tree-sitter, Cohere SDK) в подавляющем большинстве на Python. Использование TypeScript в Приложении 3 означало бы создание RAG-инфраструктуры с нуля, что не учит ничему про RAG. Использование Python во всех приложениях позволяет сосредоточиться на концепциях RAG, а не на фреймворковой обвязке.

### Почему разные фреймворки (LlamaIndex vs LangChain)?

Студенты видят, что RAG не привязан к конкретному фреймворку. LlamaIndex специализирован на RAG (меньше кода, больше магии). LangChain даёт больше контроля над каждым шагом конвейера. Знакомство с обоими помогает студентам выбирать для своих проектов. Приложение 3 возвращается к LlamaIndex, чтобы показать, как один фреймворк масштабируется от MVP до продакшена.

### Почему разные векторные БД?

Каждая представляет реальную точку принятия решения из урока:
- **ChromaDB:** нулевая настройка для прототипов (цель Приложения 1)
- **Qdrant:** специализированная векторная БД с поддержкой гибридного поиска (Приложению 2 это нужно)
- **pgvector:** путь «у вас уже есть PostgreSQL», достаточно для большинства продакшен-сценариев

### Почему разные провайдеры эмбеддингов/LLM?

Показывает, что RAG не привязан к конкретному провайдеру, и знакомит студентов с компромиссами по стоимости:
- OpenAI: самый распространённый, самый дешёвый для малых масштабов
- Cohere: мультиязычные эмбеддинги + нативное реранжирование
- Ollama: нулевая стоимость API, показывает вариант локального развёртывания
- GPT-4o-mini → GPT-4o → Claude: возрастающее качество, разное ценообразование
