# K8s Knowledge Base — Мульти-индексный RAG с оценкой и мониторингом

## Что вы построите

Полноценное RAG-приложение с тремя источниками знаний о Kubernetes:

- **Документация** — официальные markdown-страницы (концепции, задачи, туториалы)
- **Исходный код** — Go-файлы ядра Kubernetes (контроллеры, планировщик, kubelet)
- **API спецификация** — OpenAPI-описание всех эндпоинтов Kubernetes API

Приложение автоматически маршрутизирует запрос к нужным индексам, ищет по нескольким источникам параллельно, генерирует ответ с цитированием источников и отслеживает качество, задержки и стоимость.

## Чему вы научитесь

- **Мульти-индексный RAG** — раздельные индексы для разных типов данных с объединением результатов
- **Маршрутизация запросов** — LLM-классификатор направляет запрос к нужным индексам
- **Code-aware чанкинг** — tree-sitter для разбора Go-кода на функции, типы и интерфейсы
- **Оценка RAGAS** — faithfulness, answer relevancy, context relevancy, context precision
- **Семантический кэш** — pgvector для кэширования по схожести запросов
- **Трекинг стоимости и задержек** — метрики по каждому этапу пайплайна

## Архитектура

```
Запрос пользователя
        │
        ▼
   [Embedding]  ──→  [Семантический кэш]  ──→  Кэш-хит? → Ответ
        │                                           │
        ▼                                           ▼ (кэш-мисс)
   [Маршрутизатор]  ──→  Claude классифицирует запрос
        │
        ▼
   [Параллельный поиск]  ──→  docs_index / code_index / api_specs_index
        │
        ▼
   [Объединение и ранжирование]
        │
        ▼
   [Генерация ответа]  ──→  Claude с мульти-источниковым контекстом
        │
        ▼
   [Метрики + Кэш]  ──→  Запись задержек, токенов, стоимости
```

## Предварительные требования

- [uv](https://docs.astral.sh/uv/) — менеджер пакетов Python
- Python 3.11+
- Docker и Docker Compose
- Ключ Anthropic API (`ANTHROPIC_API_KEY`)
- ~8 ГБ свободного места (для Ollama модели и данных)

## Быстрый старт

### 1. Запустите инфраструктуру

```bash
docker compose up -d
```

Это запустит:
- PostgreSQL 16 с расширением pgvector (порт 5432)
- Ollama для локальных эмбеддингов (порт 11434)
- Автоматическую загрузку модели `nomic-embed-text`

### 2. Установите зависимости

```bash
uv sync
```

### 3. Настройте окружение

```bash
cp .env.example .env
# Отредактируйте .env — укажите ваш ANTHROPIC_API_KEY
```

### 4. Скачайте данные

```bash
uv run python scripts/download_docs.py       # ~50 страниц документации
uv run python scripts/download_code.py       # ~25 Go-файлов
uv run python scripts/download_api_specs.py  # OpenAPI спецификация
```

### 5. Проиндексируйте данные

```bash
uv run python index.py --source all
```

При повторном запуске индексируются только изменённые файлы. Для полной переиндексации:

```bash
uv run python index.py --source all --force
```

### 6. Запустите приложение

```bash
uv run streamlit run app.py
```

## Как это работает

### Индексация

Каждый тип данных обрабатывается по-своему:

- **Документация**: разбивается по markdown-заголовкам с сохранением иерархии секций
- **Исходный код**: tree-sitter парсит Go-файлы, извлекая функции, типы и интерфейсы как отдельные чанки
- **API спецификация**: каждый эндпоинт (метод + путь + описание) становится одним чанком

Все эмбеддинги генерируются локально через Ollama (`nomic-embed-text`) и хранятся в pgvector.

### Маршрутизация запросов

Claude анализирует вопрос и решает, в каких индексах искать:
- Концептуальные вопросы → docs
- Вопросы об имплементации → code
- Вопросы об API → api_specs
- Сложные вопросы → несколько индексов параллельно

### Семантический кэш

Перед маршрутизацией эмбеддинг запроса сравнивается с кэшем в pgvector. Если найден запрос с cosine similarity > 0.95, возвращается кэшированный ответ без обращения к LLM.

### Оценка

Тестовый датасет из 50 вопросов покрывает все три индекса. RAGAS вычисляет четыре метрики:
- **Faithfulness** — насколько ответ соответствует контексту
- **Answer Relevancy** — насколько ответ релевантен вопросу
- **Context Relevancy** — насколько контекст релевантен вопросу
- **Context Precision** — насколько точно найдены нужные фрагменты

## Оценка качества

```bash
uv run python evaluate.py
```

Результаты сохраняются в базе данных и отображаются на странице «Оценка» в UI.

## Упражнения

### 1. Добавьте новый источник данных
Добавьте четвёртый индекс — блог-посты Kubernetes (https://kubernetes.io/blog/). Создайте скрипт загрузки, добавьте индексацию и обновите маршрутизатор.

### 2. Замените модель эмбеддингов
Замените Ollama `nomic-embed-text` на OpenAI `text-embedding-3-small`. Сравните оценки RAGAS до и после замены. Как изменилась стоимость?

### 3. Измерьте эффект кэширования
Отключите семантический кэш и выполните 20 запросов. Затем включите кэш и повторите те же запросы. Сравните общую стоимость и среднюю задержку.

### 4. Добавьте порог уверенности
Если ни один найденный чанк не набрал relevance score > 0.6, вместо генерации ответа верните сообщение «Недостаточно информации для ответа на этот вопрос». Как это влияет на метрику faithfulness?
